Sedifex Integration API Guide (v1)

This guide defines how third-party websites (including Buy Sedifex) should authenticate, call endpoints, deduplicate/cache responses, and migrate safely as the API evolves.

1) Authentication and API keys

Required server-side config:

  • SEDIFEX_INTEGRATION_API_KEY (legacy: SEDIFEX_INTEGRATION_KEY)
  • SEDIFEX_API_BASE_URL (legacy: SEDIFEX_INTEGRATION_API_BASE_URL)
  • SEDIFEX_STORE_ID
SEDIFEX_API_BASE_URL=https://us-central1-sedifex-web.cloudfunctions.net
SEDIFEX_STORE_ID=store_123
SEDIFEX_INTEGRATION_API_KEY=sk_live_xxx
  • Admin key mode: can access all stores when storeId is omitted.
  • Store key mode: store-scoped access; storeId is required.
  • Never expose integration keys in browser bundles.

Required headers for authenticated endpoints:

x-api-key: <master_or_store_integration_key>
X-Sedifex-Contract-Version: 2026-04-13
Accept: application/json

2) Contract versioning

  • Current contract version: 2026-04-13.
  • Request header: X-Sedifex-Contract-Version.
  • Response headers: x-sedifex-contract-version, x-sedifex-request-id.

3) Endpoints (integration surface)

  • GET /v1/products (public marketplace feed)
  • GET /v1IntegrationProducts?storeId=<storeId> (authenticated)
  • GET /api/public-blog?storeId=<storeId>[&slug=<postSlug>] (public)
  • GET /v1IntegrationPromo?storeId=<storeId> or ?slug=<promoSlug>
  • GET /v1IntegrationAvailability?storeId=<storeId> (+ optional filters)
  • GET /integrationGallery?storeId=<storeId>
  • GET /integrationCustomers?storeId=<storeId>
  • GET /integrationTopSelling?storeId=<storeId>&days=30&limit=10
  • GET /integrationTikTokVideos?storeId=<storeId>
  • GET/POST /v1IntegrationBookings?storeId=<storeId>
  • POST /integration/checkout/create
  • GET /integration/orders/:reference
  • POST /integration/webhooks/payment-status (Sedifex outbound)

4) Integration page wiring sequence

  1. Load env config: base URL, store ID, integration key.
  2. Attach required auth + contract headers.
  3. Fetch integration endpoints server-side with ?storeId=....
  4. Normalize records and dedupe by stable ids.
  5. Cache + fallback + retry idempotent GET failures with backoff.
  6. Log x-sedifex-request-id on all failures.

5) Booking and checkout contract (important updates)

  • Create booking first via POST /v1IntegrationBookings, then create checkout session.
  • Pre-checkout naming rule: use bookingId before hosted checkout returns; do not treat pre-checkout IDs as a real sedifexOrderId.
  • On create/update of integration booking documents, write syncStatus: "pending" and syncRequestedAt.
  • Return URL completion is not payment proof; final state comes from webhook delivery or GET /integration/orders/:reference.
  • In return/confirmation UI, explicitly tell users the team has received their data and provide WhatsApp/email support channels for follow-up.

6) Checkout pricing contract

Use Sedifex as source of truth for totals (all values in minor units).

  • fulfillment_type (PICKUP | DELIVERY)
  • subtotal, tax_total, delivery_fee
  • pre_processing_total, processing_fee_to_add, final_total

Required sequence: preview → create → paystack webhook → order status fetch.

7) Caching and deduplication

  • Deduplicate by id (or productId), optionally by updatedAt.
  • Products/top-selling: 30s cache + 120s stale-while-revalidate.
  • Promo/gallery: 60s cache + 300s stale-while-revalidate.
  • Keep small local fallback datasets for transient failures.

8) Error handling

  • 400: malformed request or contract mismatch
  • 401: invalid integration key
  • 404: unknown store or slug
  • 405: unsupported method

9) Migration and compatibility

  • Additive fields may appear within same contract version; ignore unknown fields.
  • Breaking changes ship via new contract version with overlap migration window.
  • Use shared/integrationTypes.ts as shared type source of truth.
